## Goal

Use the configured shared-wiki MCP tools to inspect indexed shared wiki pages, validate proposed shared-wiki patches, and create branch + PR updates against the configured GitHub shared-wiki repository.

## Non-negotiable rules

- Treat this as a standalone adapter skill, not as a Superpowers development workflow step.
- Treat the GitHub shared-wiki repo as the shared wiki source of truth for this MCP workflow.
- Do not merge PRs from this skill.
- Do not write directly to `.shared-superpowers/wiki/` when the user is using the GitHub MCP flow.
- Do not let MCP decide whether knowledge is durable, shared-owned, or neutral; the agent must decide those before calling write tools.
- Do not publish system-specific identifiers, internal URLs, environment names, local paths, deployment instance identifiers, or current-system-only business rules to shared wiki.
- Do not mix this flow with the `publish-shared-wiki` skill; that skill is for local shared wiki submodule/repository publishing.
- Do not use this skill as the normal wiki discovery path during brainstorming or planning; use `wiki-researcher` for that.

## User Input

Use the user request that invoked this skill as the input. If the user invoked the skill explicitly with arguments, treat those arguments as the primary input.

## Expected MCP tools

The user should have copied `mcp/shared-wiki/`, configured it with a repo such as:

```text
https://github.com/YWJ-hy/shared-wiki.git
```

and enabled the MCP server in Claude Code.

Expected tools:

- `shared_wiki_status`
- `shared_wiki_tree`
- `shared_wiki_read`
- `shared_wiki_read_section`
- `shared_wiki_search`
- `shared_wiki_validate_patch`
- `shared_wiki_create_patch_pr`

If the tools are unavailable, tell the user to build and configure the MCP server from `mcp/shared-wiki/README.md`.

## Read workflow

1. Call `shared_wiki_status` first.
2. If status is healthy enough for reads, call `shared_wiki_tree`.
3. Use `shared_wiki_read` for root/directory indexes and leaf companion `xxx.index.md` files.
4. Use `shared_wiki_read_section` for selected sections from leaf `xxx.md` files; do not read whole leaf documents unless the user explicitly asks for a manual audit.
5. Use `shared_wiki_search` for a bounded keyword search when index navigation is insufficient.
6. Do not scan the whole repo unless the user explicitly asks for an audit.
7. Treat `displayPath` as a logical `.shared-superpowers/wiki/...` path, not a local file path outside this MCP workflow.

## Write / PR workflow

Before creating a patch:

1. Confirm the knowledge is durable and reusable across sibling projects.
2. Check existing indexed shared wiki pages for semantic duplicates.
3. Confirm the content belongs in shared wiki rather than project wiki.
4. Rewrite the content into neutral, portable language.
5. Check whether creating or updating pages needs user authorization under shared wiki policy.

Then:

1. Prepare a unified diff that changes only shared wiki markdown pages or allowed settings files.
2. Call `shared_wiki_validate_patch`.
3. If validation fails, fix the patch or report why it cannot proceed.
4. If validation passes and the user has authorized the write scope, call `shared_wiki_create_patch_pr`.
5. Report the returned branch, PR URL, changed files, and validation summary; do not claim the PR is merged or published.

## Completion checklist

- [ ] MCP status was checked.
- [ ] Existing indexed shared wiki pages were reviewed enough to avoid duplicates.
- [ ] The proposed content is neutral and portable.
- [ ] Required create/update authorization was satisfied.
- [ ] The patch validated successfully.
- [ ] A branch + PR was created, or the reason it could not be created was reported.
- [ ] The response did not claim the PR was merged or published.
