Generated by superpower-adapter.

This skill is a reference and fallback guide, not the default wiki selection path.

Default adapter flow:
1. During `brainstorming`, use `wiki-researcher` with `phase: brainstorm` to disclose relevant project/shared wiki context. There is no `maxWikiPages` cap; relevance is controlled by progressive index navigation, not by dropping matching wiki.
2. During `writing-plans`, use `wiki-researcher` with `phase: plan` to formally select relevant project/shared wiki candidate sections. `wiki-researcher` must not create `taskWikiRefs`, compute `taskFingerprint`, invent task IDs, or use `appliesTo` for routing.
3. Write the Superpowers implementation plan with stable task headings such as `### Task T1: <title>`. Only after source-of-truth verification, plan review, and final task stabilization may the planning flow bind selected wiki sections to tasks.
4. Read `contracts/wiki-context-v3.example.jsonc`, then write detailed selected-section constraints to `docs/superpowers/plans/<plan-stem>.wiki-context.json` as schemaVersion 3 JSON, with page-rooted `wikiPages`, one bounded `documentContext` per page, nested `sections`, section `destination`, `globalWikiRefs`, `taskWikiRefs`, and `taskFingerprint` entries for final tasks.
5. Validate the sidecar with `python3 __SUPERPOWER_ADAPTER_PLUGIN_ROOT__/scripts/wiki_context_render.py docs/superpowers/plans/<plan-stem>.wiki-context.json --validate-only --strict --execution-ready --plan-path docs/superpowers/plans/<plan-stem>.md`; do not inspect `scripts/wiki_context_render.py` to infer the authoring format.
6. Record a lightweight `## Referenced Project Wiki` section in the plan that links the `.wiki-context.json` file and summarizes selected pages, selected sections, task-bound/global/planning-only counts, hard constraints, and MCP revisions.
7. Before execution or SDD dispatch, run exactly one fingerprint preflight with `wiki_context_render.py --fingerprint-preflight --strict --execution-ready --plan-path <plan>`. If the user manually changed reviewed task text, refresh only `taskWikiRefs` and `taskFingerprint` from the current plan and already-selected sections; do not rewrite the plan, reselect wiki, or let implementers decide routing.
8. During implementation and review, consume the plan's `Referenced Project Wiki` and render selected role constraints from the linked `.wiki-context.json` with `wiki_context_render.py --task-id <stable-task-id> --role implementer|reviewer --execution-ready`. Forced hard-constraint rereads use `--task-id <stable-task-id> --reread-list` and include document context plus the selected section body only.
9. After finishing work, use the `update-wiki` skill to review whether durable implementation knowledge should persist in the appropriate wiki root.

Use this skill only when:
- the user asks how progressive wiki disclosure works;
- `wiki-researcher` is unavailable and a manual fallback is needed;
- you are debugging adapter installation or native skill patch behavior.

Reference rules:
1. Start with existing root indexes: `.superpowers/wiki/index.md` and `.shared-superpowers/wiki/index.md`.
2. For shared wiki, use the selected source consistently: local `.shared-superpowers/wiki/` or the configured GitHub-backed shared-wiki MCP source.
3. Follow links or listed paths to narrower indexes before reading leaf wiki pages.
4. Read only the detailed wiki sections needed for the current phase; use per-document section indexes to carry page title/overview once as page-level context for selected sections.
5. Do not read either wiki tree in full unless the human explicitly asks for a full audit. Unlimited wiki means no page cap, not permission for broad unfocused scans.
6. If one root index is missing, continue with the other root and mention the caveat.
7. If both root indexes are missing, report that project/shared wiki is unavailable instead of guessing paths.
8. Treat adapter sidecar JSONL files as diagnostics only, not as the required source of truth for new work.
9. When shared wiki comes from MCP, `displayPath` is logical; use `wikiPath` plus revision metadata when rereading.
10. Do not use legacy `appliesTo` to route wiki constraints to tasks.

Manual fallback flow:
- Understand task.
- Read existing root indexes: `.superpowers/wiki/index.md` first, then `.shared-superpowers/wiki/index.md`.
- Follow links or listed paths to narrower indexes inside each root.
- Read the minimum relevant leaf wiki sections; if only a section is needed, include companion index title/overview as bounded context, not sibling sections or whole-page body.
- Use root-prefixed paths such as `.shared-superpowers/wiki/frontend/contracts.md` in selected-page output.
- If planning, read `contracts/wiki-context-v3.example.jsonc`, write selected constraints to `docs/superpowers/plans/<plan-stem>.wiki-context.json`, wait until final task headings exist, bind sections through `taskWikiRefs` / `globalWikiRefs`, validate with `wiki_context_render.py --validate-only --strict --execution-ready --plan-path <plan>`, and link it from `Referenced Project Wiki`.
- If implementing or reviewing, use the plan's `Referenced Project Wiki` and linked `.wiki-context.json`; render task-scoped constraints with `wiki_context_render.py --task-id <stable-task-id>`, and for hard-constraint rereads, inject document context before the selected section body and do not include sibling sections or the whole page.
- Never synthesize task mappings during execution fallback. If task routing is missing or stale, stop before code changes and perform the planning-side refresh described above.
- After finishing work, update relevant wiki pages if new knowledge should persist.
