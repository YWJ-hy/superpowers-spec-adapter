Generated by superpower-adapter.

Use this skill to turn a reusable engineering practice into a **layered skill pack**, or to restructure an existing skill into that shape, and to register a lightweight **wiki discovery card** so future planning can find and bind it.

This is an adapter authoring skill. Its completion proves only that a skill pack was scaffolded/converted and its discovery card registered; it does not prove any implementation, review, or plan task is complete. Do not invoke Superpowers completion/verification skills because this skill ran.

## When to use

- A reusable practice keeps getting re-explained (management-page layout, micro-app host/child file structure, form/table conventions, a fixed review checklist) and should become an executable skill.
- An existing monolithic `SKILL.md` is large and loads in full for both implement and review — restructure it so heavy content loads on demand.
- `update-wiki` handed off a **skill-pack candidate** (reusable workflow/process knowledge that belongs in an executable skill, not a wiki leaf page).

## Pack model (one fixed file, open beyond that)

The only invariant is that **`SKILL.md` is a thin router**: frontmatter (`name` + a trigger-rich `description`), when-to-use, mode selection, a few core hard rules, and pointers to sibling files. Everything else is an **open set** created only when the content warrants it:

```
.claude/skills/<name>/
  SKILL.md          # required: thin router
  implement.md      # optional (common): implementation flow
  review.md         # optional (common): review checklist
  rules.md          # optional (common): full normative rules
  examples.md       # optional (common): examples
  scripts/  references/  templates/  assets/  ...   # any other files/dirs
```

**Why only SKILL.md is fixed**: progressive disclosure — keep SKILL.md small so an implement task pulls implement content and a review task pulls review content, instead of loading everything every time; a short `description` also matches more reliably. The full content still lives in the pack, so the skill works outside Superpowers.

Do not invent a fixed five-file template. Split by what the content actually needs (a pure generator may have no `review.md`; a small skill stays small).

## Framework-independent authoring

Author each pack so it works with or without Superpowers. In `SKILL.md`, prefer this context priority rather than hard-coding any wiki path:

1. project rules already present in the current context;
2. a wiki/doc/spec path the user provides;
3. locally discoverable project docs;
4. this skill's generic flow.

If project-specific rules are missing, ask for the relevant practice document before making irreversible design choices. **Do not maintain a "which wiki to read" routing table inside the skill** — the wiki points at the skill, not the other way around.

## Mechanical helper

All file operations, discovery-card registration, companion-index regeneration, index linkage, and validation are mechanical and belong to the script. You do the semantic work (naming, the `description`, trigger scenarios, deciding what content goes into which file, writing the prose).

```bash
# create a new pack skeleton (open file set; only SKILL.md is mandatory)
python3 __SUPERPOWER_ADAPTER_PLUGIN_ROOT__/scripts/scaffold_practice_skill.py \
  --project-root . --json scaffold --name <skill-name> \
  --description "<trigger-rich one-liner>" --files implement.md,review.md,rules.md

# convert an existing skill (NON-DESTRUCTIVE): stages a new pack, carries over all
# bundled files (scripts/, assets), and reports source content not yet represented
python3 __SUPERPOWER_ADAPTER_PLUGIN_ROOT__/scripts/scaffold_practice_skill.py \
  --project-root . --json convert --from <path/to/old/SKILL.md-or-dir> \
  --name <skill-name> --files rules.md,review.md

# register / update the wiki discovery card + companion index + index linkage
python3 __SUPERPOWER_ADAPTER_PLUGIN_ROOT__/scripts/scaffold_practice_skill.py \
  --project-root . --json register-card --name <skill-name> \
  --title "<card title>" --triggers "<scenario keywords>" [--authorized-create|--authorized-update]

# validate pack invariants + discoverability (+ optional source coverage)
python3 __SUPERPOWER_ADAPTER_PLUGIN_ROOT__/scripts/scaffold_practice_skill.py \
  --project-root . --json validate --name <skill-name> [--from <old SKILL.md-or-dir>]
```

## Workflow

### Create

1. Confirm the project root and that `.superpowers/wiki/index.md` exists.
2. Choose a kebab-case `name`, a trigger-rich `description`, and decide which sibling files the content actually needs (the open set).
3. Run `scaffold` to write the skeleton, then author the real content into `SKILL.md` and each file. Keep `SKILL.md` thin; push heavy rules/examples into siblings.
4. Run `register-card` (see authorization below) to create the discovery card and make it discoverable.
5. Run `validate --name <name>` and fix any errors.

### Convert (non-destructive)

1. Run `convert --from <existing skill>`; it stages a new pack, carries over every bundled file (including `scripts/`), and lists `uncovered` source content.
2. Author the layered files so every `uncovered` heading/section is represented somewhere; never drop content. Bundled non-markdown files are already carried over.
3. Run `validate --name <name> --from <existing skill>` until there are no coverage gaps.
4. Show the user the new pack and a diff against the original. **Replace the original only after the user confirms** (the script never deletes the original). If the staged pack was written to `<name>.converted/`, move it into place on confirmation.
5. Run `register-card`.

## Authorization

`register-card` writes to the project wiki and respects the selected root's `wiki.updateAuthorization` policy (`.superpowers/settings.json`). Defaults: `createNewDocument` = `ask`, `updateExistingPage` = `skip`. When the policy is `ask`, get explicit user authorization first, then pass `--authorized-create` (first card / new `skills.md`) or `--authorized-update` (existing `skills.md`). When it is `refuse`, do not write; report it.

## Boundaries

- Do not copy or replace the official `skill-creator`. The user may start a new skill with `skill-creator`; this skill normalizes it into a layered pack and registers the discovery card.
- Do not move the practice's normative content into the wiki. The wiki holds only the discovery card (`guides/skills.md`); the pack holds the content.
- Convert is non-destructive: never delete or overwrite the original skill without explicit user confirmation, and never lose a bundled file.

## Completion checklist

- [ ] `SKILL.md` is a thin router with a trigger-rich `description`; heavy content lives in sibling files.
- [ ] Only the files the content needs were created (open set, not a forced template).
- [ ] For convert: all source headings/sections are represented and all bundled files (incl. `scripts/`) were preserved; the original was replaced only after user confirmation.
- [ ] The discovery card was registered under the policy, and `validate` reports the pack discoverable with no errors.
- [ ] The pack is usable outside Superpowers (context-priority phrasing, no hard-coded wiki routing table).
