# Break the Loop

Use this after a bug has been fixed and verified, especially when the bug repeated, crossed component boundaries, required multiple failed attempts, exposed an implicit assumption, or revealed a test coverage gap.

This is an after-action review. It does not replace Superpowers `systematic-debugging`. If the bug is not fixed, the root cause is unclear, or verification is incomplete, stop this retrospective and return to `systematic-debugging` first.

---

## Relationship to Other Skills

Normal bug workflow:

```text
systematic-debugging
→ break-loop
→ update-wiki, only when durable knowledge should persist
```

- `systematic-debugging` finds the root cause, tests hypotheses, implements the fix, and verifies it.
- `break-loop` analyzes the completed debugging work and extracts prevention lessons.
- `update-wiki` is the only place that persists durable implementation knowledge into `.superpowers/wiki/`.

Do not use this skill to choose wiki targets, check semantic duplicates, edit `.superpowers/wiki/`, or refresh indexes. Hand candidate knowledge to `update-wiki` instead.

## Preconditions

Before starting, confirm:

- [ ] The bug or failing behavior has been reproduced or otherwise evidenced.
- [ ] The root cause has been identified.
- [ ] The final fix has been implemented.
- [ ] The fix has been verified with tests, commands, or a documented manual check.

If any item is missing, report what is missing and use `systematic-debugging` before continuing.

## Analysis Framework

### 1. Debugging Evidence

Summarize the evidence from the completed debugging process:

- **Symptom**: What failed or behaved unexpectedly?
- **Expected behavior**: What should have happened?
- **Actual behavior**: What happened instead?
- **Reproduction signal**: How was the failure observed or reproduced?
- **Root cause hypothesis**: What cause was confirmed?
- **Evidence gathered**: What proved the root cause?
- **Failed attempts**: What fixes or hypotheses did not work, if any?
- **Final fix**: What changed?
- **Verification**: What proved the fix worked?

### 2. Root Cause Category

Classify the bug into the strongest matching category.

| Category | Characteristics | Example |
|---|---|---|
| **A. Missing Wiki Guidance** | No durable guidance existed for how to implement safely | New behavior without a checklist or project rule |
| **B. Cross-Layer Contract** | The interface between layers was unclear or violated | API, CLI, file, event, or config shape differed across layers |
| **C. Change Propagation Failure** | One place changed but dependent places were missed | Signature, path, state, schema, or convention changed without updating callers |
| **D. Test Coverage Gap** | Existing checks missed the integrated behavior | Unit passed, installed command or end-to-end path failed |
| **E. Implicit Assumption** | Code relied on an undocumented assumption | Timing, default value, environment, ordering, units, or ownership was assumed |

### 3. Why Fixes Failed

Include this section only when there were failed attempts.

For each failed attempt, identify the failure mode:

- **Surface Fix**: Fixed the symptom, not the source.
- **Incomplete Scope**: Found part of the root cause but missed related paths.
- **Tool Limitation**: Search, type checks, tests, or logs did not expose the real boundary.
- **Mental Model**: Investigation stayed in the wrong layer or assumed the wrong ownership.
- **Architecture Signal**: Three or more failed fixes suggested the pattern itself needed reconsideration.

### 4. Prevention Mechanisms

Identify what would prevent this class of bug from recurring.

| Type | Description | Example |
|---|---|---|
| **Documentation** | Write durable guidance so future sessions know the rule | Add a contract or gotcha to project wiki |
| **Architecture** | Make the unsafe path impossible or harder | Centralize ownership, remove ambiguous state, use typed boundaries |
| **Compile-time** | Let tooling catch the mistake earlier | Type signatures, schema checks, strict config validation |
| **Runtime** | Detect or surface the issue at execution time | Assertions, logs, diagnostics, health checks |
| **Test Coverage** | Verify the integrated behavior that failed | Smoke, regression, end-to-end, install-path, or command-path test |
| **Code Review** | Add a review prompt for known risk | Checklist item for cross-layer contract changes |

### 5. Systematic Expansion

Look beyond the single fixed bug:

- **Similar issues**: Where else might the same mistake exist?
- **Design flaw**: Does this reveal an unstable ownership boundary or architecture pattern?
- **Process flaw**: Did the workflow miss an expected check?
- **Knowledge gap**: What did the debugging process reveal that future sessions should know earlier?

### 6. Update-Wiki Handoff Decision

Decide whether the retrospective produced durable implementation knowledge.

Good handoff candidates include:

- project-specific rules or contracts;
- validation and error behavior;
- gotchas or common mistakes;
- cross-layer checklists;
- design decisions with rationale;
- testing requirements for an integrated path.

Skip `update-wiki` when the finding is only:

- a temporary incident note;
- an obvious fact already visible from code;
- a one-off debugging timeline;
- an unverified guess;
- a formatting-only or typo-only issue.

## Handoff Rules

If the decision is `skip update-wiki`, stop after reporting the analysis.

If the decision is `run update-wiki`, use the `update-wiki` skill next and pass the handoff packet as input context. Do not edit `.superpowers/wiki/` from this skill.

`update-wiki` must still make its own decisions: whether each candidate is durable, how to split candidates atomically, which indexed wiki pages to read, whether the idea is already covered, which leaf wiki page owns the rule, and how to refresh and validate the wiki index.
