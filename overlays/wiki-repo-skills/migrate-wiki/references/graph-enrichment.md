# migrate-wiki › Graph Enrichment (mode 2)

Load this only for **mode 2**. It re-organizes a wiki that is already in section-marker
format into a typed knowledge graph: page node types + `[[page#section]]` knowledge edges.
This is a **batched, propose-then-confirm** pass — never a silent mass edit, and never a
commit (the parent skill is edit-only).

Resolve the toolchain as in the parent skill:

```bash
WIKI_DIR="${CLAUDE_PROJECT_DIR:-$(git rev-parse --show-toplevel 2>/dev/null || pwd)}"
TOOLS="$WIKI_DIR/.claude/wiki-tools/scripts"
```

## Non-negotiables

- Do **not** change the semantic content of pages. Node types and edges are additive
  metadata (a frontmatter `type:` line and `[[ ]]` links inside section bodies).
- Respect the wiki's authorization policy: editing existing pages is governed by
  `wiki.updateAuthorization.updateExistingPage` in `.shared-superpowers/settings.json`
  (default `skip` = proceed after the proposal is confirmed, `ask` = get explicit
  approval, `refuse` = do not write). If the file is absent, treat it as `skip` but still
  confirm each batch.
- Do not invent relationships. Only add an edge when two sections genuinely relate; only
  set a non-default type when the page truly is domain/decision/guide.
- Work in **bounded batches** (one directory subtree, or ~5–10 pages at a time). Confirm
  each batch before writing, so a large wiki stays reviewable and the pass is resumable.

## Step A — Assign node types

For each page in the batch, decide its node `type:`:

| type | use for |
|---|---|
| `constraint` | executable coding rules/contracts (the default — leave frontmatter off) |
| `domain` | stable domain/business facts implementation must respect (glossaries, rules) |
| `decision` | ADR-style decisions with rationale and rejected alternatives |
| `guide` | cross-cutting checklists / thinking prompts |

Most pages stay `constraint`. Only propose a non-default type when the page's primary
content matches. Set it as a leading frontmatter block:

```markdown
---
type: decision
---

# <Page Title>
```

## Step B — Author knowledge edges

For each section, identify genuine cross-references to other sections and add edges in the
section body. Page paths are **wiki-root-relative** (relative to the repo root):

- `[[page#section]]` — `see-also` (related context); `[[#section]]` is same-page;
  `[[page]]` is whole-page.
- `[[depends-on: page#section]]` — this section's constraint is incomplete without the
  target. **Load-bearing**: a `depends-on` target of a hard-constraint section is auto-reread
  (1-hop) by consumers. Use sparingly and only when the target is truly required.
- `[[supersedes: page#section]]` — this section replaces the target (newer decision wins).
- `[[contradicts: page#section]]` — unresolved conflict; surface, do not resolve here.
- Colon must be followed by a space. Only `[[ ]]` is an edge — plain `(path#anchor)` is not.

## Step C — Present the batch proposal

Show a reviewable proposal for the batch and **wait for confirmation before writing**:

```
Batch: backend/ (4 pages)

Page types:
  backend/api/contract.md      → constraint (unchanged)
  backend/glossary.md          → domain      (was constraint)
  backend/decisions/cursor.md  → decision    (was constraint)

Proposed edges:
  backend/api/contract.md#response-format
      → depends-on backend/api/contract.md#error-envelope   (errors must follow the envelope)
  backend/decisions/cursor.md#choice
      → supersedes backend/decisions/offset.md#choice        (cursor replaces offset paging)
```

Let the user confirm, adjust, or drop items. Only write the confirmed set.

## Step D — Write, regenerate, lint

1. Write the confirmed frontmatter types and `[[ ]]` edges into the pages (subject to the
   authorization policy above).
2. Regenerate companion indexes + `.graph.json` (this fills `引用` / `被引用` columns,
   the `> Type:` line, and `pageTypes`):

   ```bash
   python3 "$TOOLS/wiki_migrate_helper.py" --generate-indexes "$WIKI_DIR" --wiki-dir "$WIKI_DIR"
   ```

3. Lint for dangling edges and unknown types, and fix what it reports before the next batch:

   ```bash
   python3 "$TOOLS/wiki_update_check.py" --wiki-dir "$WIKI_DIR" --json
   ```

   `.graph.json`'s `dangling` array lists unresolved `[[ ]]` targets and unknown edge types;
   `wiki_update_check.py` also warns on unknown page `type:` values.

Repeat A–D per batch. Report per batch: pages typed, edges added, dangling/unknown remaining.
Leave committing to the user.
