Generated by superpower-adapter.

Use this skill when you are working **inside a standalone wiki repository** (wiki content at the repo root) and want to (re)structure existing pages: convert prose pages to the section-marker format, and optionally weave a typed `[[page#section]]` knowledge graph across the whole wiki.

This is a structural migration of the local repo. It is edit-only and leaves committing to you.

## Resolve the toolchain

```bash
WIKI_DIR="${CLAUDE_PROJECT_DIR:-$(git rev-parse --show-toplevel 2>/dev/null || pwd)}"
TOOLS="$WIKI_DIR/.claude/wiki-tools/scripts"
```

All commands pass `--wiki-dir "$WIKI_DIR"`, treating the repo root as the wiki root directly.

## Modes

When invoked, **first ask the user which mode to run** (do not assume):

1. **Sections only** — convert pages to the two-layer section-marker format and generate companion indexes. Follow the Workflow below and stop.
2. **Sections + graph enrichment** — do mode 1 for any page still lacking section markers, then re-organize the wiki into a typed knowledge graph (page node `type:` + `[[page#section]]` edges). After the Workflow below, load and follow `references/graph-enrichment.md`.

Mode 2 is a superset of mode 1. A wiki already in section-marker format skips straight to enrichment.

## Non-negotiable rules

- **Edit-only. Never run `git add`, `git commit`, `git push`, or open a PR.** Report what changed and leave committing to the user.
- Do **not** change the semantic content of pages — only add section markers (and, in mode 2, additive `type:` frontmatter and `[[ ]]` edges) around existing content. Do not delete, reorder, or rewrite existing text.
- Ask the user to confirm the migration plan before applying changes.
- A section must be an **independently referenceable constraint unit**, not an arbitrary paragraph.
- Section IDs are kebab-case (`[a-z0-9][a-z0-9_-]*`) and reflect the constraint's core semantics.
- This wiki is shared/portable: do not introduce system-specific identifiers, internal URLs, environment names, local paths, or current-system-only business rules.

## Section Design Principles

1. **One section = one independently citable constraint unit.** If a future reader needs rule X but not rule Y from the same page, split them.
2. **Merge related headings.** "规则" + "示例" + "常见错误" describing one constraint → ONE section.
3. **Split compound headings** that contain multiple independent constraints.
4. **Section ID = semantic key** (`path-based-update`, not `section-2`).
5. **Nesting for hierarchy** when a broad topic has independently citable sub-topics.
6. **Keep `hard` sections tight** around the rule plus its do/don't examples; move only compliance-irrelevant background into the overview or a `soft` section.

## Workflow

### Step 1 — Inventory

```bash
python3 "$TOOLS/wiki_migrate_helper.py" --inventory "$WIKI_DIR" --wiki-dir "$WIKI_DIR"
```

Lists every indexed leaf page with line counts, heading structure, and whether it already has section markers.

### Step 2 — Analyze each document

For each leaf page: read it fully, identify semantically independent constraint/knowledge blocks, decide section boundaries and IDs, write a one-line semantic overview (not a mechanical copy of the first paragraph), and classify each section's strength (`hard` for mandatory rules, `soft` for recommendations/patterns).

### Step 3 — Present the migration plan

Show a per-document summary (overview + numbered sections with line ranges and strength) and ask the user to confirm or adjust before writing.

### Step 4 — Apply markers and overview

For each confirmed document:

1. Insert `<!-- wiki-section:xxx -->` / `<!-- /wiki-section:xxx -->` around the identified sections. On each opening marker, author a one-line **summary** (an abstraction of the section's theme, NOT a comma-listed enumeration of every rule — a list re-states the body and must be re-edited whenever a rule is added): `<!-- wiki-section:xxx summary="一句话总结本节主题与核心约束" -->`. Strongly target ≤ 140 chars (one tight line; 140 is not a hard cap, the index shows it verbatim); single line, no `"`/`>`/newlines. The `wiki-researcher` reads this during brainstorm; without it the index falls back to a front-biased mechanical excerpt. Marker metadata, not a rewrite of body prose.
2. Keep the page's `# Title` in the `.md` file.
3. Remove document-level overview/blockquote/`## 概览` summary text from the `.md` file.
4. Create or update the companion `<stem>.index.md` header with the confirmed overview:

   ```markdown
   # Sections: <relative-path-to-page.md>

   > <semantic overview of the document's topic and scope>
   ```

   The section table below the header is generated in the next step.

### Step 5 — Validate and generate

```bash
python3 "$TOOLS/wiki_migrate_helper.py" --validate "$WIKI_DIR" --wiki-dir "$WIKI_DIR"
python3 "$TOOLS/wiki_migrate_helper.py" --generate-indexes "$WIKI_DIR" --wiki-dir "$WIKI_DIR"
python3 "$TOOLS/wiki_update_check.py" --wiki-dir "$WIKI_DIR" --json
```

The index generator updates only the section table and preserves the `.index.md` header (including the overview), and rebuilds `.graph.json`.

### Step 6 — Report (do not commit)

Summarize: documents processed, sections created, companion indexes generated/updated, and the validation result. Tell the user to review the diff and commit themselves. Do not run git write commands.

## Backfilling summaries on already-sectioned pages

A page that already has section markers skips the marker-insertion steps, so its sections may
predate the `summary="…"` attribute. Backfill them **mechanically** — read only the bodies that
lack a summary, never re-scan whole wiki documents:

1. List the sections missing a summary, **with their bodies**, in one call:

   ```bash
   python3 "$TOOLS/wiki_migrate_helper.py" --missing-summaries "$WIKI_DIR" --wiki-dir "$WIKI_DIR" --with-body --json
   ```

   Output is `[{root, path, sectionId, body}]` — exactly the sections to summarize, nothing else.

2. For each, write a one-line **summary** (theme-level abstraction, not a rule list). Confirm the
   proposed summaries with the user before writing.

3. Write them back in one batch — JSONL of `{path, sectionId, summary}` on stdin:

   ```bash
   python3 "$TOOLS/wiki_migrate_helper.py" --set-summaries "$WIKI_DIR" --wiki-dir "$WIKI_DIR" < summaries.jsonl
   ```

4. Regenerate indexes + graph (`--generate-indexes`, as in the Workflow).

This reads each missing-summary body exactly once and loads no unrelated pages into context.
