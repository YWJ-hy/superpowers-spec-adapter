Generated by superpower-adapter.

Use this skill when you are working **inside a standalone wiki repository** (the wiki content lives at the repo root — `index.md`, topic directories, leaf pages) and you want to add or revise durable knowledge directly.

This is the **author/curator side** of a shared wiki. It is NOT the consumer flow: it does not review a finished implementation task, read execution sidecars, or open a GitHub MCP PR. It edits the local repo's wiki pages and leaves committing to you.

## Resolve the toolchain

The vendored mechanical helpers were stamped into this repo by `superpower-adapter export-wiki-skills`. Resolve them once per session:

```bash
WIKI_DIR="${CLAUDE_PROJECT_DIR:-$(git rev-parse --show-toplevel 2>/dev/null || pwd)}"
TOOLS="$WIKI_DIR/.claude/wiki-tools/scripts"
```

All commands below pass `--wiki-dir "$WIKI_DIR"`, which treats the repo root as the wiki root directly.

## Non-negotiable rules

- **Edit-only. Never run `git add`, `git commit`, `git push`, or open a PR.** Stop after regenerating indexes and validating; report what changed and let the user review and commit.
- Do the semantic work yourself. The Python helpers are mechanical only — they never decide whether knowledge is durable, where it belongs, or whether it is neutral.
- This wiki is shared/portable. Do **not** write system-specific identifiers, internal URLs, environment names, local filesystem paths, deployment instance identifiers, or current-system-only business rules. Rewrite knowledge into neutral, portable terms before saving.
- Detailed rules live in leaf pages, never in `index.md`. Keep indexes lightweight.
- Respect the wiki's own authorization policy (see *Authorization*).

## Wiki First Rule

Capture **durable, reusable engineering knowledge**, not a summary of one change. Default to NOT writing. Promote only knowledge that passes a test:

- executable rules reused across modules/pages/workflows/layers
- API / payload / command contracts stable outside one call site
- validation and error behavior future work must preserve
- design decisions with rationale (hard to reverse + surprising + a real trade-off)
- stable domain / business facts implementation must respect
- cross-cutting thinking checklists
- non-obvious gotchas future readers would otherwise rediscover wrongly

Skip local business logic, details obvious from code, and anything that belongs in a spec/PR/commit message.

Each leaf page declares a node `type:` in frontmatter — `constraint` (default, executable rules), `domain` (stable domain facts), `decision` (ADR-style, usually with `supersedes`/`contradicts` edges), or `guide` (cross-cutting checklists). Set the matching `type:` when you create a page.

## Workflow

### 1. Decide and target

1. Decide there is durable knowledge worth persisting. If not, say so and stop — change nothing.
2. Read the relevant existing pages before writing (start from `index.md` and the topic directory's `index.md`, then the leaf `xxx.index.md` companion). Do not scan the whole tree.
3. Check for a semantic duplicate. Prefer editing the owning page over creating a near-duplicate.
4. Choose the owning leaf page (or decide a new page is justified) and its node `type:`.

### 2. Edit the page

- Wrap each independently citable constraint unit in section markers, and author a one-line
  `summary="…"` on the opening marker:

  ```markdown
  <!-- wiki-section:kebab-case-id summary="一句话，说明本节约束什么" -->
  ## Heading
  ...rule + its do/don't examples...
  <!-- /wiki-section:kebab-case-id -->
  ```

- Section IDs are kebab-case (`[a-z0-9][a-z0-9_-]*`) and reflect the constraint's semantics, not a heading translation.
- The `summary="…"` is what the `wiki-researcher` reads during brainstorm (only the index table is read then). Distill the section's load-bearing point, not its title; one line, ≤ ~100 chars, no `"`/`>`/newlines. Omitting it falls back to a front-biased mechanical excerpt, so prefer authoring it for every non-trivial section.
- Keep a `hard` section (mandatory rules: 必须/禁止/MUST/MUST NOT/REQUIRED/SHALL NOT) tight around the rule together with its examples — those get reread in full at execution.
- When you create a new page, give it a `# Title` and, if non-default, a leading `--- type: ... ---` frontmatter block.
- For a brand-new page, also add it to the owning directory's `index.md` so it becomes discoverable. List it inside the `<!-- superpower-adapter:auto:start -->` / `:auto:end` block; the regeneration step below refreshes the one-line summaries.

### 3. Maintain knowledge edges (incremental)

When the edited section genuinely relates to another section, add an edge inside the section body. Page paths are wiki-root-relative:

- `[[page#section]]` — `see-also` (related context); `[[#section]]` same-page; `[[page]]` whole-page.
- `[[depends-on: page#section]]` — this section is incomplete without the target (load-bearing; used sparingly).
- `[[supersedes: page#section]]` — this section replaces the target.
- `[[contradicts: page#section]]` — unresolved conflict; surface, don't resolve here.

The colon must be followed by a space. Only `[[ ]]` is an edge; a plain `(path#anchor)` link is not. For a wholesale re-typing / edge-weaving pass over the whole wiki, use the `migrate-wiki` skill (mode 2) instead.

### 4. Regenerate and validate

```bash
# Rebuild the edited page's companion index + the repo's .graph.json:
python3 "$TOOLS/wiki_generate_section_index.py" "<relative/path/to/page.md>" --wiki-dir "$WIKI_DIR"

# Refresh directory index summaries + the section graph across the whole repo:
python3 "$TOOLS/wiki_migrate_helper.py" --generate-indexes "$WIKI_DIR" --wiki-dir "$WIKI_DIR"

# Validate mechanics, neutrality guards, and dangling [[ ]] edges:
python3 "$TOOLS/wiki_update_check.py" --wiki-dir "$WIKI_DIR" --json
```

Fix any `INVALID` errors (unclosed fences, marker errors, neutrality violations) before finishing. Resolve dangling section-link warnings.

### 5. Report (do not commit)

Summarize: which pages were created/edited, sections added, indexes/`.graph.json` regenerated, and the validation result. Then tell the user to review the diff and commit themselves — for example `git status` / `git diff`, then commit and push (or open a PR branch) at their discretion. Do not run those commands yourself.

## Authorization

This repo's write policy lives in `.shared-superpowers/settings.json` under `wiki.updateAuthorization` (`updateExistingPage` default `skip`, `createNewDocument` default `ask`; values `skip` / `ask` / `refuse`) and `wiki.sharedNeutrality` (`blockedTerms` / `blockedPatterns` — mechanical neutrality guards enforced by `wiki_update_check.py`). If the file is absent, neutrality guards are empty (no mechanical blocklist) — you still apply neutrality judgment, and you may suggest the user add the file. Honor `refuse` by not writing.
