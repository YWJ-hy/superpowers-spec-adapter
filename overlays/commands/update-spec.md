When you learn something valuable from implementation, debugging, review, or discussion, use this command to update the relevant spec documents under `.superpowers/spec/`.

**Timing**: After completing a task, fixing a bug, discovering a new pattern, or making a design decision that future sessions should know.

---

## Spec First Rule

In this adapter, spec content should capture **durable implementation knowledge**, not generic prose.

Good spec updates include:
- executable rules
- API / payload / command contracts
- validation and error behavior
- design decisions with rationale
- project conventions
- gotchas and common mistakes
- cross-cutting thinking checklists in `guides/`

Do not treat `index.md` files as the main place for detailed rules. Keep detailed content in leaf spec files and keep indexes lightweight.

---

## When to Update Specs

| Trigger | Example | Target Spec |
|---|---|---|
| Implemented a feature | Added a new request flow | Relevant `backend/*.md` or `frontend/*.md` |
| Fixed a bug | Found non-obvious error handling behavior | Relevant leaf spec such as `backend/error-handling.md` |
| Made a design decision | Chose one contract shape over another | Relevant leaf spec + design-decision section |
| Found a reusable pattern | Better way to structure components or APIs | Relevant leaf spec |
| Found a gotcha | Learned that X must happen before Y | Relevant leaf spec or `guides/*.md` |
| Need a checklist | Something people should remember to consider | `guides/*.md` |

---

## How to Decide Where It Belongs

Use this rule:

- **This is how to implement safely** → `backend/*.md` or `frontend/*.md`
- **This is what to think about before implementing** → `guides/*.md`

Examples:
- request/response contract shape → backend spec
- UI component state rule → frontend spec
- release readiness reminder → guide
- migration review checklist → guide

## Quality Checklist

Before finishing the update:

- [ ] Is the content durable and reusable?
- [ ] Did you choose the right target spec file?
- [ ] Is the content specific instead of generic?
- [ ] Did you explain **why**, not just **what**?
- [ ] Did you avoid putting detailed rules into `index.md`?
- [ ] Did you refresh the index chain after updating body content?

## Execution Layer

### Preferred one-shot path

If you already know the hint, title, why, and rules:

```bash
python3 superpowers/scripts/spec_update_run.py "error handling" "Error normalization" "Prevent inconsistent backend error shapes." "Normalize backend error payloads" "Keep user-facing messages stable"
```

This will:
1. choose the target spec file
2. append a structured update block to the body
3. refresh the index chain

### Staged path

If you want more control:

```bash
python3 superpowers/scripts/spec_select_target.py "error handling"
python3 superpowers/scripts/spec_update_prompt.py <target-spec-relative-path>
python3 superpowers/scripts/spec_update_template.py <target-spec-relative-path> "Title" "Why it matters"
python3 superpowers/scripts/spec_apply_update.py <target-spec-relative-path> "Title" "Why it matters" "Rule 1" "Rule 2"
python3 superpowers/scripts/update-spec.py
```

## Notes

- Only `index.md` at the spec root is mandatory.
- The rest of the directory structure is project-defined.
- Detailed spec files should stay separate from entry indexes so the system can load them progressively.
- By default, `draft/`, `archive/`, and `examples/` directories are ignored by index generation.
- Add more ignored directory names in `.superpowers/spec/.adapter-ignore` using one directory name per line.
