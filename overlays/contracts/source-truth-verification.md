# Adapter Source-of-Truth Verification (configured / requested only)

Generated by superpower-adapter.

Read and follow this contract only when `writing-plans` has determined that source-of-truth
verification applies — that is, `source_truth_settings.py --show-policy` reported a configured
`sourceOfTruth`, or the user explicitly requested source-of-truth verification. When `status` is
`not_configured` and no explicit request was made, skip source-of-truth entirely and do not create
a sidecar; you do not need this contract in that case.

This detail intentionally lives outside the always-loaded `writing-plans` patch so unconfigured
plans never pay for it. Do not inline this whole contract back into the planning skill.

## 1. Dispatch the verifier

After the complete draft plan contains concrete implementation assumptions, dispatch the
`source-of-truth-verifier` agent. Provide:

- the draft plan path,
- the linked `docs/superpowers/plans/<plan-stem>.wiki-context.json` path if one exists,
- `.superpowers/settings.json`,
- the output paths `docs/superpowers/plans/<plan-stem>.source-truth-report.json` and
  `docs/superpowers/plans/<plan-stem>.source-truth-constraints.json`.

The verifier checks configured `sourceOfTruth.sources` only; `sourceOfTruth.heuristics` defaults
false. The constraints sidecar is always written; the full report is written only when there is
something to audit (status `needs_revision` / `blocked`, or non-empty findings). On a clean pass the
report is skipped and you get a single file.

## 2. Consume only the bounded envelope

The verifier writes the constraints sidecar itself (and the full report only when there are findings
to audit) and returns only a bounded verdict envelope (status, output paths, summary counts,
constraintSet titles, blocking deltas, short caveats); it does not return the full report contents
into your context. When the report was skipped, `outputReportPath` is null and no report file exists
— do not try to open it. The full report is planning/audit only, not normal execution context; do
not read it during planning unless you must resolve a specific finding. If the verifier reports it
could not write the files (`filesWritten: false`), it returns each file's JSON inline — write those
files yourself and confirm they exist on disk before continuing.

## 3. Bind execution constraints after task stabilization

The constraints sidecar is a two-phase artifact: the verifier writes only the raw `constraintSets`
and `status`, with execution routing left unbound. After verifier-driven revisions and final task
stabilization, read the verifier-written `docs/superpowers/plans/<plan-stem>.source-truth-constraints.json`
back from disk and bind execution constraints by assigning each set a `destination` and writing
`globalConstraintRefs`, `taskConstraintRefs`, and `taskFingerprint`. Do not use legacy `appliesTo`,
task-title strings, or draft task IDs for execution routing. Validate once a constraints sidecar
exists:

```bash
python3 __SUPERPOWER_ADAPTER_PLUGIN_ROOT__/scripts/source_truth_render.py docs/superpowers/plans/<plan-stem>.source-truth-constraints.json --validate-only --strict --execution-ready --plan-path docs/superpowers/plans/<plan-stem>.md
```

## 4. Record a lightweight plan section

Add a lightweight `## Source-of-Truth Verification` section to the plan only when verification ran.
Include status, constraints sidecar path, truth source count, blocking count, binding status, and
short caveats; do not paste the full report. If verifier status is `blocked`, do not offer execution
handoff.
