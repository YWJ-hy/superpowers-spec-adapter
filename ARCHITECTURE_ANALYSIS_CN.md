# superpower-adapter 架构分析

本文整理当前 `superpower-adapter` 的两个核心问题：

1. 它如何对 `.superpowers/spec/` 做渐进式披露
2. 它如何介入到 Superpowers 的运行链路中

---

## 一、总体结论

可以先用一句话概括：

> `superpower-adapter` 不是去改造 Superpowers 的核心工作流，而是通过安装 overlay 文件、补充 SessionStart hooks、增加 skills / commands / scripts，把一个“spec 渐进式披露 + plan sidecar 上下文复用”的子系统挂接到 Superpowers 上。

换句话说：

- **Superpowers** 负责主方法论和会话入口
- **superpower-adapter** 负责项目 spec 的渐进式读取、索引维护、以及 plan sidecar 上下文复用

---

## 二、superpower-adapter 如何做 spec 的渐进式披露

核心策略不是把 `.superpowers/spec/` 全量加载进上下文，而是分成四步：

1. 会话启动时只注入轻量摘要树
2. 规划时先读 `index.md`，再逐层下钻到更窄的 index
3. 只按需读取当前任务相关的 leaf spec
4. 完成任务后再通过 `update-spec` 维护索引链

### 2.1 启动时只注入摘要树

入口在 `overlays/hooks/session-spec-index`。

关键逻辑：

- 调用 `python3 .../scripts/spec-context.py --tree --depth 2`
- 将结果包成 `<spec-index>...</spec-index>`
- 作为 SessionStart 的额外上下文注入

对应文件：

- `superpower-adapter/overlays/hooks/session-spec-index:13`
- `superpower-adapter/overlays/scripts/spec-context.py:57`

这意味着在会话一开始，模型拿到的不是 spec 正文，而是一个最多两层的 spec 摘要树。

### 2.2 摘要树由 spec-context.py + spec_common.py 生成

`spec-context.py` 提供三种模式：

- `--tree`：输出递归摘要树
- `--category`：读取某个分类目录的 `index.md`
- `--file`：读取具体 leaf spec 文件

对应文件：

- `superpower-adapter/overlays/scripts/spec-context.py:58-62`

底层摘要逻辑在 `spec_common.py`：

- 默认忽略 `draft`、`archive`、`examples`
- 允许通过 `.adapter-ignore` 扩展忽略目录
- 摘要提取规则是“第一层标题 + 第一条非空正文”

对应文件：

- `superpower-adapter/overlays/scripts/spec_common.py:8`
- `superpower-adapter/overlays/scripts/spec_common.py:23`
- `superpower-adapter/overlays/scripts/spec_common.py:47`

因此，启动时注入的 `spec-index` 本质上是一个轻量导航图，而不是完整规则文本。

### 2.3 逐步下钻规则由 skill 约束

渐进披露不只靠 hook，还靠 skill 明确约束 agent 的读取方式。

在 `overlays/skills/spec-progressive-disclosure/SKILL.md` 中规定：

1. 先读 `.superpowers/spec/index.md`
2. 顺着 index 链走向更窄的 index 或 leaf spec
3. 只读取当前任务最小必要的文件
4. 不要读取整个 `.superpowers/spec/` 树，除非用户明确要求

对应文件：

- `superpower-adapter/overlays/skills/spec-progressive-disclosure/SKILL.md:10-16`

这说明当前设计里，“渐进式披露”既有运行时注入，也有行为层约束。

### 2.4 update-spec 维护可导航索引链

为了让后续能够稳定地下钻，adapter 会维护一套自动索引链。

核心脚本是 `overlays/scripts/update-spec.py`，负责：

- 确保 `.superpowers/spec/index.md` 存在
- 给有内容的子目录补 `index.md`
- 在自动区块中生成：
  - child indexes
  - leaf files
- 保留自动区块外的人工内容

对应文件：

- `superpower-adapter/overlays/scripts/update-spec.py:19-22`
- `superpower-adapter/overlays/scripts/update-spec.py:44-54`
- `superpower-adapter/overlays/scripts/update-spec.py:57-91`
- `superpower-adapter/overlays/scripts/update-spec.py:94-107`

自动维护的区块边界是：

```markdown
<!-- superpower-adapter:auto:start -->
<!-- superpower-adapter:auto:end -->
```

因此，spec 的渐进式披露能够成立，前提就是：

- spec 目录被组织成一个可导航的 index 链
- index 只存导览，不承载过多细节
- 细节沉淀在 leaf spec 中

### 2.5 这一部分的实际调用链

可以压缩成一条链：

```text
SessionStart
  -> session-spec-index
  -> spec-context.py --tree --depth 2
  -> spec_common.py 生成摘要树
  -> 注入 <spec-index>
  -> 后续 agent 按 skill 约束从 index.md 逐层下钻
```

## 三、superpower-adapter 如何介入到 Superpowers 的运行链路

adapter 的接入方式不是直接改 Superpowers 核心，而是采用 overlay 安装。

### 3.1 adapter 是一个 overlay 包

`manifest.json` 声明了要安装到目标 `superpowers/` 目录中的内容：

- skills
- commands
- scripts
- hooks

对应文件：

- `superpower-adapter/manifest.json:7-22`

包含的关键增强点有：

- `skills/spec-progressive-disclosure/SKILL.md`
- `skills/plan-context-sidecar/SKILL.md`
- `commands/update-spec.md`
- `commands/plan-context.md`
- `scripts/spec-context.py`
- `scripts/update-spec.py`
- `scripts/plan-context.py`
- `hooks/session-spec-index`
- `hooks/session-plan-context`

### 3.2 install.sh 负责复制 overlay 并 patch hooks

安装入口在 `install.sh`。

它做了两件关键事：

1. 把 overlay 文件复制到目标 Superpowers 目录
2. 调 `lib/hook_patch.py install` 修改 hook 配置

对应文件：

- `superpower-adapter/install.sh:28-48`
- `superpower-adapter/install.sh:64-70`

同时它有一个安全措施：

- 如果目标文件存在，但不包含 `Generated by superpower-adapter` 标记，就拒绝覆盖

对应文件：

- `superpower-adapter/install.sh:41-44`

### 3.3 hook_patch.py 把 adapter hooks 挂到 SessionStart

`lib/hook_patch.py` 会往 SessionStart 里追加两个 hook：

- `session-spec-index`
- `session-plan-context`

对应文件：

- `superpower-adapter/lib/hook_patch.py:10-17`
- `superpower-adapter/lib/hook_patch.py:28-45`

安装后的结果可以在 `superpowers/hooks/hooks.json` 中看到：

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|clear|compact",
        "hooks": [
          { "command": "... session-start" },
          { "command": "... session-spec-index" },
          { "command": "... session-plan-context" }
        ]
      }
    ]
  }
}
```

对应文件：

- `superpowers/hooks/hooks.json:3-23`

这说明 adapter 的主要介入点是：

- **会话启动 hook 节点**
- 而不是底层执行引擎节点

## 四、SessionStart 时三层上下文如何叠加

当前运行时，进入模型上下文的内容可以视为三层叠加。

### 4.1 第一层：Superpowers 原生方法论层

来源：`superpowers/hooks/session-start`

它会把 `using-superpowers` skill 全文注入上下文，告诉 agent：

- 你已经有 superpowers
- 其他 skill 应通过 Skill tool 使用
- 当前会话的基础方法论是什么

对应文件：

- `superpowers/hooks/session-start:17-35`

### 4.2 第二层：spec 导航层

来源：`session-spec-index`

它注入 `<spec-index>`，内容是 `.superpowers/spec/` 的轻量摘要树，只负责告诉 agent：

- 有哪些 spec
- 有哪些 index / leaf 文件
- 它们的大致摘要是什么

对应文件：

- `superpower-adapter/overlays/hooks/session-spec-index:13-27`

### 4.3 第三层：当前计划状态层

来源：`session-plan-context`

它调用 `plan-context.py current`，如果当前存在 `.superpowers/current-plan`，就注入：

- 当前 plan 是哪个文件
- sidecar 在哪里
- 当前 phase 是什么
- 如何 render implement / review 上下文

对应文件：

- `superpower-adapter/overlays/hooks/session-plan-context:13-31`
- `superpower-adapter/overlays/scripts/plan-context.py:202-220`

### 4.4 三层叠加后的职责分工

可以总结为：

| 层 | 来源 | 作用 |
|---|---|---|
| 方法论层 | `session-start` | 告诉 agent 如何在 Superpowers 中工作 |
| spec 导航层 | `session-spec-index` | 告诉 agent 项目规则在哪里 |
| 当前计划层 | `session-plan-context` | 告诉 agent 当前任务已有上下文可复用 |

因此，adapter 并没有取代 Superpowers 的主流程，而是在 SessionStart 上追加了两个与 spec 和 plan 相关的辅助上下文层。

## 五、plan sidecar 是如何工作的

adapter 除了 spec 渐进披露外，还增加了一套 plan sidecar 机制，用来复用规划阶段已经选好的上下文。

### 5.1 plan markdown 仍然是主计划文件

`plan-context-sidecar` skill 明确要求：

- `docs/superpowers/plans/*.md` 仍然是主计划文档
- 不把计划迁移成目录型结构

对应文件：

- `superpower-adapter/overlays/skills/plan-context-sidecar/SKILL.md:10-13`

### 5.2 sidecar 目录保存上下文选择结果

每个 plan 对应一个 sidecar 目录：

```text
docs/superpowers/plans/<stem>.context/
```

其中包含：

- `plan.jsonl`
- `implement.jsonl`
- `review.jsonl`
- `state.json`

路径规则见：

- `superpower-adapter/overlays/scripts/plan-context.py:61-63`
- `superpower-adapter/overlays/scripts/plan-context.py:81-86`

### 5.3 current-plan 作为当前计划指针

当前激活的计划通过 `.superpowers/current-plan` 表示。

对应文件：

- `superpower-adapter/overlays/scripts/plan-context.py:15`
- `superpower-adapter/overlays/scripts/plan-context.py:88-104`

这个文件起到的是当前会话工作状态指针的作用，而不是长期文档。

### 5.4 规划阶段把选中的 spec 写入 sidecar

规划阶段，agent 按最小必要原则选 spec，然后执行：

- `plan-context.py add --phase plan ...`

把记录写入 `plan.jsonl`。

每条记录包含：

- `path`
- `reason`
- `phase`
- `mode`
- `selectedBy`
- `createdAt`

对应文件：

- `superpower-adapter/overlays/scripts/plan-context.py:278-307`

### 5.5 实现 / 评审阶段优先复用 sidecar

实现阶段：

- 优先使用 `plan.jsonl + implement.jsonl`

评审阶段：

- 优先使用 `plan.jsonl + review.jsonl`

只有在 sidecar 缺失、无效或不足时，才回退到重新发现 `.superpowers/spec`。

对应约束：

- `superpower-adapter/overlays/skills/plan-context-sidecar/SKILL.md:14-19`
- `superpower-adapter/overlays/skills/spec-progressive-disclosure/SKILL.md:11-15`

### 5.6 render 阶段支持 summary / full 两种展开模式

`plan-context.py render --phase ...` 会把记录 materialize 成当前可用上下文。

当记录为：

- `mode=summary`：只输出摘要
- `mode=full`：直接展开 spec 正文

对应文件：

- `superpower-adapter/overlays/scripts/plan-context.py:136-160`
- `superpower-adapter/overlays/scripts/plan-context.py:310-343`

所以“渐进”不只是对 spec 树本身，也体现在 plan sidecar 的 materialize 过程中。

## 六、任务结束后如何把新知识写回 spec

任务完成后，adapter 提供 `update-spec` 工作流，把新的 durable knowledge 写回 `.superpowers/spec/`。

### 6.1 一键入口

`spec_update_run.py` 会串起三个动作：

1. 选择目标 spec
2. 写入结构化更新块
3. 刷新索引链

对应文件：

- `superpower-adapter/overlays/scripts/spec_update_run.py`

### 6.2 结构化更新块写入 leaf spec

设计上不鼓励把详细规则写进 `index.md`，而是写到 leaf spec 中。

对应命令文档：

- `superpower-adapter/overlays/commands/update-spec.md:26`
- `superpower-adapter/overlays/commands/update-spec.md:166-169`

这样做的目的就是：

- `index.md` 保持轻量导航
- 细节沉淀在 leaf spec
- 系统下次仍然能轻量索引、按需展开

### 6.3 刷新索引链后，下次 SessionStart 自动反映

因为 `update-spec.py` 会更新 index 的自动区块，所以新的 leaf spec 或摘要在下一次 SessionStart 时，会自动出现在 `<spec-index>` 摘要树中。

这形成了一个闭环：

```text
实现 / 调试 / 设计决策
  -> update-spec
  -> 写 leaf spec
  -> update-spec.py 刷新 index
  -> 下次 SessionStart 注入新的摘要树
```

## 七、把整体时序压成一张图

```text
[SessionStart]
  -> session-start
     -> 注入 using-superpowers
  -> session-spec-index
     -> spec-context.py --tree --depth 2
     -> 注入 <spec-index>
  -> session-plan-context
     -> plan-context.py current
     -> 注入 <current-plan-context>

[Planning]
  -> 从 .superpowers/spec/index.md 开始
  -> 顺着 index 逐层下钻
  -> 选择最少必要 spec
  -> plan-context.py add --phase plan
  -> 写入 plan.jsonl

[Implementation]
  -> render(plan.jsonl + implement.jsonl)
  -> 不足时才回源到 .superpowers/spec

[Review]
  -> render(plan.jsonl + review.jsonl)
  -> 不足时才回源到 .superpowers/spec

[After Task]
  -> update-spec
  -> 写 leaf spec
  -> update-spec.py 刷新 index 链
  -> 下次 SessionStart 自动看到更新后的摘要
```

## 八、最终结论

### 8.1 关于 spec 渐进式披露

当前 adapter 的 spec 渐进式披露机制由三部分共同构成：

1. `session-spec-index` 在启动时注入轻量摘要树
2. `spec-progressive-disclosure` skill 约束 agent 从 `index.md` 逐层下钻
3. `update-spec.py` 维护一个稳定的 index 链，保证目录始终可导航

### 8.2 关于介入 Superpowers 的方式

adapter 不是直接修改 Superpowers 核心，而是通过：

- overlay 安装
- SessionStart hook patch
- 新增 skills / commands / scripts

把 spec 子系统和 plan sidecar 子系统挂接到 Superpowers 的主流程旁边。

### 8.3 更准确的定义

如果要给这个架构下一个更准确的描述，可以说：

> `superpower-adapter` 是一个面向 Superpowers 的增强型 overlay，它通过 SessionStart 上下文注入、spec 索引链维护、以及 plan sidecar 复用机制，为项目级实现规则提供渐进式披露和可复现的任务上下文选择。
